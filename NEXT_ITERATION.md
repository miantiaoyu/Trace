# 下一迭代计划

## 迭代目标

在保持低频、只读查询的前提下，扩大已验证路线覆盖率，并让统一入口在真实数据库样本上稳定运行。

## 需求范围

- 为统一入口增加请求超时、有限重试和单船司限速配置。
- 验证最近 7 天真实数据量，确认默认 20 条样本是否适合日常运行。
- 为 COSCO、ONE、MSC、万海的页面结构变化增加更细粒度的回归样本。
- 继续复查 HMM，并在不绕过人机验证的前提下评估 CMA CGM、OOCL、ZIM 等受限路线。
- 评估是否需要把统一 JSON 报告转换成后续落库所需的领域字段，但暂不改变只读原则。

## 不在本轮范围

- 不写公司数据库。
- 不保存本地 JSON 文件。
- 不绕验证码、不破解加密参数、不绕过登录或访问控制。
- 不做可视化界面。
- 不做高频批量请求。

## 验收标准

- 统一入口能读取最近 7 天订单，并为每条记录选择正确路线。
- 已验证船司能返回原始数据，单条失败不影响后续记录。
- 受限船司能返回明确的 `route_unavailable` 或 `query_failed` 状态。
- 结果报告包含数据库样本元数据和成功/失败汇总。
- `prod-db.yml` 不被修改、不被提交。
- 公司数据库只发生读取，不发生写入。

## 建议实现步骤

1. 用 `--limit 1` 对每个已验证船司各跑一条最新样本。
2. 根据真实返回内容确认路线的事件覆盖范围和异常类型。
3. 增加有限重试、限速和结果摘要策略。
4. 再决定是否进入字段解析或落库设计。
