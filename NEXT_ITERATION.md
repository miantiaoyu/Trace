# 下一迭代计划

## 迭代目标

在 Linux 服务器环境验证批量运行稳定性，并建立马士基与 HMM 的长期回归和运行观测基线。

## 需求范围

- 在 Linux 服务器环境验证 Docker 镜像中的 HMM/万海 Xvfb 运行方式，并记录浏览器依赖和资源基线。
- 根据 `TRACKING_STAGE_VALIDATION_REPORT.md` 制作脱敏固定测试夹具，优先覆盖马士基、HMM、ONE、阳明和 MSC 的时间语义。
- 为长荣寻找发船前样本，为万海寻找发船前和明确在途样本；没有真实样本时保持字段缺口，不人工推断。
- 用最近订单分批运行马士基与 HMM，统计成功率、平均耗时、重试率和失败类型。
- 根据批量结果校准马士基与 HMM 的限速、超时和瞬时故障重试策略。
- 在服务器定时任务中以 Compose 一次性容器启用脱敏运行日志和船司连续失败状态，验证任务锁、告警阈值与日志轮转策略。
- 评估是否需要把统一 JSON 报告转换成后续落库所需的领域字段，但暂不改变只读原则。
- 当单轮规模超过约 200 个柜或需要任务断点恢复时，再评估引入 Crawlee 持久队列；当前规模不进行框架迁移。

## 不在本轮范围

- 不写公司数据库。
- 不保存本地 JSON 文件。
- 不绕验证码、不破解加密参数、不绕过登录或访问控制。
- 不做可视化界面。
- 不做高频批量请求。
- 不扩展新的船司路线。

## 验收标准

- Linux + Xvfb 环境能连续执行 HMM 查询，无需人工点击。
- 马士基与 HMM 批量结果包含成功率、耗时、重试次数和失败类型统计。
- 两次定时任务重叠时，后启动的任务明确退出，不会并发访问船司官网。
- 脱敏日志和健康状态不包含柜号、原始响应或数据库凭证，连续失败达到阈值时产生明确告警。
- 单柜超时、解析失败和页面结构变化均不会中断后续柜号。
- 结果报告包含数据库样本元数据和成功/失败汇总。
- `prod-db.yml` 不被修改、不被提交。
- 公司数据库只发生读取，不发生写入。

## 建议实现步骤

1. 在 Linux + Xvfb 环境验证 HMM 单条与连续查询。
2. 对马士基和 HMM 各运行一组低频真实订单样本并统计指标。
3. 补充脱敏回归样本和异常分类测试。
4. 根据统计结果校准默认策略。
