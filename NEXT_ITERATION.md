# 下一迭代计划

## 迭代目标

集中加固马士基与 HMM 两条高订单量路线，使当前状态、船名航次和 ETA 在真实订单批量查询中可验证、可诊断。

## 需求范围

- 马士基根据 `event_time_type` 区分实际与预计事件，并明确当前状态的选择规则，不依赖官网数组顺序。
- 为马士基和 HMM 增加脱敏固定样本回归测试，覆盖未装船、在途、转运和到港阶段。
- 为 HMM 增加明确的部分数据、无轨迹、页面契约变化和瞬时访问失败分类。
- 根据生产批量数据调整马士基与 HMM 的限速、超时和瞬时故障重试策略。
- 在 Linux 服务器环境验证 HMM 的 Xvfb 运行方式，并记录浏览器依赖和资源基线。
- 评估是否需要把统一 JSON 报告转换成后续落库所需的领域字段，但暂不改变只读原则。

## 不在本轮范围

- 不写公司数据库。
- 不保存本地 JSON 文件。
- 不绕验证码、不破解加密参数、不绕过登录或访问控制。
- 不做可视化界面。
- 不做高频批量请求。
- 不扩展其他船司路线。

## 验收标准

- 统一入口能读取最近订单并正确路由马士基与 HMM。
- 两家船司均返回原始数据和固定摘要，单条失败不影响后续记录。
- HMM 页面结构变化不能被误报为成功。
- 结果报告包含数据库样本元数据和成功/失败汇总。
- `prod-db.yml` 不被修改、不被提交。
- 公司数据库只发生读取，不发生写入。

## 建议实现步骤

1. 固化马士基 `ACTUAL/EXPECTED` 事件语义和当前状态规则。
2. 扩充 HMM 不同运输阶段的脱敏回归样本与异常分类。
3. 在 Linux + Xvfb 环境验证 HMM 批量运行。
4. 根据真实批量结果校准限速、超时和重试策略。
