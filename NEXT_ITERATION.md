# 下一迭代计划

## 迭代目标

在 Linux 服务器环境验证批量运行稳定性，并建立马士基与 HMM 的长期回归和运行观测基线。

## 需求范围

- 在公司 Linux 测试服务器验证已修复的 HMM/万海 Xvfb 运行方式，并记录浏览器依赖和资源基线。
- 根据 `TRACKING_STAGE_VALIDATION_REPORT.md` 制作脱敏固定测试夹具，优先覆盖马士基、HMM、ONE、阳明和 MSC 的时间语义。
- 为长荣寻找发船前样本，为万海寻找发船前和明确在途样本；没有真实样本时保持字段缺口，不人工推断。
- 用最近订单分批运行马士基与 HMM，统计成功率、平均耗时、重试率和失败类型。
- 根据批量结果校准马士基与 HMM 的限速、超时和瞬时故障重试策略。
- 对上一轮 `query_failed` 记录运行一次到期重查，统计 HMM 恢复率和成功回写后的业务字段覆盖率。
- 继续观测万海 Incapsula 拦截；只在标准官网表单恢复可用时验证路线，不绕过官网访问控制。
- 在本机 Docker 环境先完成“两个月新单窗口 + 不限制数量”的测试库落库回归，确认 `oms.headway` 字段映射、任务锁、脱敏运行日志和船司连续失败状态。
- 在测试环境通过 XXL-JOB `SHELL` 任务调用 Docker wrapper，验证 XXL 重试、退出码和脱敏任务日志。
- 正式上线前单独评审目标数据库权限与配置，将 `target_db` 从内网测试 `oms` 切换为正式 `oms`，保持阿里 ERP 源连接只读。
- 在测试库低峰期执行 `sql/headway_add_comments.sql`，并用 `SHOW FULL COLUMNS` 确认表和全部字段的中文备注已生效。
- 验证统一 JSON 到 `oms.headway` 字段的映射和目的港实际卸船判定，不写 ERP 源表。
- 在测试库 `oms` 回归稳定后，单独评审结果落库方案；未经字段、权限和保留周期确认，不创建写入表。
- 当单轮规模超过约 200 个柜或需要任务断点恢复时，再评估引入 Crawlee 持久队列；当前规模不进行框架迁移。

## 不在本轮范围

- 不修改 ERP 源表；仅允许写入独立的 `oms.headway` 结果表。
- 不保存本地 JSON 文件。
- 不绕验证码、不破解加密参数、不绕过登录或访问控制。
- 不做可视化界面。
- 不做高频批量请求。
- 不扩展新的船司路线。

## 验收标准

- Linux + Xvfb 环境能连续执行 HMM 查询，无需人工点击。
- 马士基与 HMM 批量结果包含成功率、耗时、重试次数和失败类型统计。
- 两次定时任务重叠时，后启动的任务明确退出，不会并发访问船司官网。
- 脱敏日志和健康状态不包含柜号、原始响应或数据库凭证，连续失败达到阈值时产生明确告警。
- 单柜超时、解析失败和页面结构变化均不会中断后续柜号。
- 结果报告包含数据库样本元数据和成功、部分成功、跳过、失败汇总。
- `prod-db.yml` 不被修改、不被提交。
- ERP 源表只读；`--persist` 只对 `oms.headway` 执行受控 upsert。
- Docker/XXL-JOB 运行日志不保存柜号或官网原始响应，脱敏指标可以独立查看。
- 联调任务只读阿里正式 ERP，只向内网测试 `oms.headway` 写入；命令行和环境变量不能反转读写方向。
- 失败记录到期后能再次入选；成功重查后同一拼柜号仍只有一条记录，`query_status`、原始 JSON、统一字段和错误字段均反映最新结果。
- `SHOW FULL COLUMNS FROM oms.headway` 显示所有字段备注非空，且表中原有数据条数与内容不变。

## 建议实现步骤

1. 在本机 Docker 中执行 `--days 60 --limit 0 --persist` 的测试库落库回归。
2. 对马士基和 HMM 各运行一组低频真实订单样本并统计指标。
3. 在测试 XXL-JOB 中执行 `deploy/xxl-job/trace-test.sh`，验证退出码和重试行为。
4. 补充脱敏回归样本和异常分类测试，再单独评审正式 `oms` 目标配置。
